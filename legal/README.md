## ⛔ Чего НЕЛЬЗЯ делать без юриста

1. **Менять юридический смысл** — формулировки обязательств, прав субъекта ПД, срока хранения, правовых оснований обработки, ссылок на законы (152-ФЗ, 149-ФЗ, ст. 18.1, ст. 19 и т.д.)
2. **Удалять разделы** из Политики или Согласия
3. **Расширять перечень обрабатываемых ПД** без одновременного обновления формы на лендинге и бэкенда (`api/lead.js`). Перечень в документах и в реальности **обязаны совпадать** — расхождение это нарушение ФЗ-152
4. **Публиковать биометрию, специальные категории ПД, финансовые данные** — документы под это не написаны
5. **Менять реквизиты ИП** без подтверждения от Наиля — ЕГРИП всегда первоисточник
6. **Делать чекбокс согласия pre-checked** — по ФЗ-152 и GDPR согласие должно быть активным действием
7. **Убирать серверную валидацию `body.consent === true` в `api/lead.js`** — клиентскую проверку обойти тривиально, сервер это единственный источник правды для юр-аудита

## Версионирование согласия (критично)

В `main.js` в обработчике submit формы есть строка:
```js
data.consent_text_version = '2025-10-01';
```

**Эта версия записывается в Airtable** (поле `Consent text version`) для каждого лида — чтобы можно было доказать, под какой редакцией документа человек дал согласие.

**Когда менять версию:**
- Любая правка **смыслового текста** Политики или Согласия → бампнуть дату на дату правки (формат `YYYY-MM-DD`)
- Чисто косметические правки (вёрстка, опечатки вне юр-формулировок) → можно не бампать
- Добавление нового поля в форму → **обязательно** бампать

**Где синхронизировать версию:**
1. `main.js` — `data.consent_text_version = 'YYYY-MM-DD'`
2. `legal/privacy-policy.html` — `<p class="legal-meta">Действует с DD.MM.YYYY</p>`
3. `legal/consent.html` — `<p class="legal-meta">Действует с DD.MM.YYYY</p>`
4. Не забудь обновить cache buster `?v=` в `index.html` после правок `main.js`

Старые версии документов для аудита хранятся в git history — отдельного архива не делаем.

## Чекбокс согласия в форме

Лежит в `index.html` внутри `#leadForm` перед кнопкой `.lead-submit`. Структура:

```html
<label class="form-consent">
    <input type="checkbox" name="consent" id="leadConsent">
    <span class="consent-box" aria-hidden="true"></span>
    <span class="consent-text">
        Нажимая «Отправить», я соглашаюсь с
        <a href="legal/privacy-policy" target="_blank" rel="noopener">политикой обработки персональных данных</a>
        и даю
        <a href="legal/consent" target="_blank" rel="noopener">согласие на обработку персональных данных</a>.
    </span>
</label>
```

**Правила:**
- `checked` атрибут **запрещён** (согласие = активное действие)
- `required` атрибут **не ставить** — на форме `novalidate`, валидация живёт в `main.js`
- Ссылки **всегда** `target="_blank" rel="noopener"` и **без** `.html` в href
- Визуал: стили в `style.css` в секции `/* ===== Consent checkbox ===== */`. Квадратный 20×20, чёрная рамка 2px, при отметке — оранжевый фон с белой галочкой через `::after`

## Бэкенд — `api/lead.js`

Серверная валидация согласия — **не удалять**:
```js
if (body.consent !== true) {
  return res.status(400).json({ error: 'Требуется согласие на обработку персональных данных' });
}
```

Проверяется **строго `=== true`**, не truthy. Причина: клиент может прислать `"on"`, `"true"`, `1` — всё это должно отвергаться, чтобы исключить случайную отправку без явного согласия.

Бэкенд записывает в Airtable таблицу `Leads` (base `appwANdQ0Txe6BRsy`) **5 специальных полей** для юр-аудита:

| Поле Airtable | Тип | Источник |
|---|---|---|
| `Consent given` | checkbox | всегда `true` при успешной валидации |
| `Consent timestamp` | dateTime (ISO, Europe/Moscow) | `body.consent_timestamp` — клиентское время отправки формы |
| `Consent text version` | singleLineText | `body.consent_text_version` — версия из `main.js` |
| `IP` | singleLineText | `req.headers['x-forwarded-for']` (первый IP в цепочке) |
| `User agent` | multilineText | `req.headers['user-agent']` |

Эти поля созданы в Airtable через MCP `mcp__Airtable__create_field` и **не должны удаляться**. Если переименовываешь поле в Airtable — синхронизируй ключ в `api/lead.js` одновременно, иначе запись в Airtable начнёт 422-ить.

Telegram-уведомление содержит строку `✅ Согласие на ОПД получено (v...)` — это удобно для визуального контроля, но юр-источник правды это Airtable.

## Куда копать если что-то сломалось

- **Форма отправляет без галочки** → баг в `main.js`, проверь порядок: валидация consent **до** `submitBtn.disabled = true`
- **Airtable 422 `UNKNOWN_FIELD_NAME`** → переименовали поле в Airtable, в `api/lead.js` осталось старое имя. Ключи в payload должны **совпадать символ-в-символ** с именами полей (включая пробелы: `'Consent given'`, а не `Consent_given`)
- **В Airtable пустое `Consent timestamp`** → клиент не прислал `consent_timestamp`, сервер fallback-нул на свой `new Date().toISOString()` — это штатно, но если массово, значит `main.js` сломан
- **Полиси страница не открывается по `/legal/privacy-policy`** → проверь `vercel.json` что `cleanUrls: true` остался, проверь что файл реально `legal/privacy-policy.html` (не `.htm`, не вложен глубже)
- **Браузер показывает старую версию** → cache buster не обновили
